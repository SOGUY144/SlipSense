## 🔥 หมวด 1: คำถามเกี่ยวกับปัญหาและกลุ่มเป้าหมาย (Customer Value Fit — 30%)

### ❓ "ปัญหานี้มันเป็นปัญหาจริงหรือเปล่า? มีหลักฐานอะไรยืนยัน?"
> **ตอบ:** "มีครับ จากข้อมูลของ สสว. ปี 2567 ไทยมี SME ถึง 3.25 ล้านราย ซึ่งส่วนใหญ่เป็น Micro SME ที่ทำทุกอย่างด้วยตัวเอง ไม่มีเวลาหรือเงินจ้างนักบัญชี นอกจากนี้เรายังลงพื้นที่สัมภาษณ์ร้านค้าจริง พบว่าส่วนใหญ่ไม่สามารถบอกได้ว่าเดือนที่แล้วกำไรหรือขาดทุนกี่บาท พวกเขาใช้วิธี 'ดูเงินเหลือในบัญชี' เป็นตัวตัดสินใจเท่านั้นครับ"

### ❓ "ทำไมแม่ค้าไม่ใช้แอปบัญชีที่มีอยู่แล้ว เช่น FlowAccount หรือ PEAK?"
> **ตอบ:** "แอปเหล่านั้นออกแบบมาสำหรับนักบัญชีหรือบริษัทนิติบุคคลครับ มีศัพท์อย่าง เดบิต เครดิต ภาษีหัก ณ ที่จ่าย ที่ทำให้แม่ค้าตลาดหรือร้านอาหารเล็กๆ งงและเลิกใช้ไป **และที่สำคัญคือผู้ใช้ยังต้องมานั่งคีย์ตัวเลขเข้าไปเองทีละรายการอยู่ดี** ซึ่งเป็น Friction หลักที่ทำให้คนเลิกใช้ SlipSense แก้ตรงจุดนี้ เพราะผู้ใช้ไม่ต้องพิมพ์อะไรเลยแม้แต่ตัวเดียวครับ"

### ❓ "ผู้ใช้กลุ่มนี้จะยอมใช้แอปจริงๆ หรือ? เขาอาจจะขี้เกียจ"
> **ตอบ:** "เราออกแบบให้ง่ายที่สุดเท่าที่จะเป็นไปได้ครับ ขั้นตอนคือแค่ 'ถ่ายรูปสลิป → กดยืนยัน → จบ' ง่ายเหมือนโพสต์รูปลง IG เลยครับ ไม่มี Learning Curve เพราะไม่ต้องเรียนรู้ศัพท์บัญชีใดๆ ครับ"

## 🔥 หมวด 2: คำถามเกี่ยวกับ AI (AI Application — 20%)

### ❓ "AI ในโปรเจกต์นี้ทำอะไรบ้าง? ถ้าไม่มี AI จะเป็นยังไง?"
> **ตอบ:** "AI เป็นหัวใจหลักของแอปเลยครับ ถ้าไม่มี AI ผู้ใช้ต้องมานั่งพิมพ์ตัวเลขเข้าไปเองทุกครั้ง ซึ่งก็ไม่ต่างจากจดลง Excel AI ของเราทำงาน 3 อย่าง: 
> 1. **อ่านสลิป (Vision):** ใช้ gpt-4o-mini อ่านรูปภาพดึงข้อมูล ยอดเงิน วันที่ ชื่อผู้โอน/ผู้รับ ออกมาเป็น JSON
> 2. **จัดหมวดหมู่:** AI วิเคราะห์บริบทว่าเป็นรายรับหรือรายจ่าย และจัดเข้าหมวดที่เหมาะสม (ค่าวัตถุดิบ ค่าเช่า รายได้ขาย ฯลฯ)
> 3. **วิเคราะห์ Insights:** เมื่อมีข้อมูลสะสม AI จะวิเคราะห์แนะนำจุดรั่วไหลเป็นตัวเลขจริงๆ ครับ"

### ❓ "ใช้ AI Model อะไร? ทำไมถึงเลือกตัวนี้?"
> **ตอบ:** "ใช้ **OpenAI gpt-4o-mini** ผ่าน **Vercel AI SDK** ครับ เลือกตัวนี้เพราะ:
> 1. รองรับ Vision (อ่านรูปภาพได้) และ Structured Outputs (ส่งผลลัพธ์เป็น JSON ที่มีโครงสร้างชัดเจน)
> 2. ราคาประหยัด เหมาะกับ SME ที่ต้องสแกนสลิปวันละหลายใบ
> 3. เราใช้ Vercel AI SDK เป็น Abstraction Layer ดังนั้นถ้าในอนาคตมี Model ที่ดีกว่า เราเปลี่ยนได้ทันทีโดยไม่ต้องแก้โค้ดมากครับ"

### ❓ "AI อ่านสลิปแม่นยำแค่ไหน? ถ้าอ่านผิดล่ะ?"
> **ตอบ:** "จากการทดสอบกับสลิปดิจิทัลของธนาคารหลักในไทย ความแม่นยำอยู่ในระดับสูงมากครับ แต่เราก็ยอมรับว่ามี Edge cases ที่อ่านพลาดได้ เช่น สลิปที่ถ่ายเบลอ หรือสลิปที่มีรูปแบบแปลก **ดังนั้นเราจึงออกแบบระบบ Review ก่อนบันทึก** ที่แสดงระดับความมั่นใจ (High/Medium/Low) ให้ผู้ใช้เห็นว่าช่องไหนควรตรวจเป็นพิเศษ ผู้ใช้กดแก้ไขได้ก่อนกดบันทึกจริงครับ"

### ❓ "Structured Outputs คืออะไร? ทำไมต้องใช้?"
> **ตอบ:** "ปกติ AI จะตอบเป็นข้อความยาวๆ แต่ Structured Outputs ทำให้เราบังคับให้ AI ตอบเป็น JSON ที่มีโครงสร้างแน่นอน เช่น ยอดเงินต้องเป็นตัวเลข วันที่ต้องเป็นรูปแบบที่กำหนด ทำให้เราเอาข้อมูลไปเก็บในฐานข้อมูลได้ทันทีโดยไม่ต้อง Parse ข้อความเพิ่มครับ"

## 🔥 หมวด 3: คำถามเกี่ยวกับ Technical (Prototype & Feasibility — 15%)

### ❓ "Tech Stack ที่ใช้มีอะไรบ้าง?"
> **ตอบ:**
> - **Frontend:** Next.js (React) + Tailwind CSS + shadcn/ui
> - **Backend & Database:** Supabase (PostgreSQL) + Drizzle ORM
> - **AI:** Vercel AI SDK เชื่อม OpenAI gpt-4o-mini
> - **Authentication:** Phone OTP ผ่าน Supabase Auth (Login ด้วยเบอร์มือถือ ไม่ต้องจำรหัสผ่าน)
> - **Deployment:** Vercel

### ❓ "ทำไมเลือก Supabase? ทำไมไม่ใช้ Firebase?"
> **ตอบ:** "Supabase ใช้ PostgreSQL ซึ่งเป็น Relational Database ที่เหมาะกับข้อมูลธุรกรรมการเงินที่มีความสัมพันธ์กันซับซ้อน (เช่น ธุรกรรม → หมวดหมู่ → ผู้ใช้) มากกว่า Firebase ที่เป็น NoSQL ครับ นอกจากนี้ Supabase ยังมี Row Level Security (RLS) ที่ช่วยป้องกันข้อมูลระดับแถวได้ เหมาะกับข้อมูลการเงินที่ต้องการความปลอดภัยสูงครับ"

### ❓ "Prototype นี้ทำงานจริงได้แค่ไหน? หรือเป็นแค่ Mockup?"
> **ตอบ:** "เป็น Prototype ที่ทำงานได้จริง 100% ครับ ไม่ใช่ Mockup:
> - เชื่อม AI API จริง (OpenAI) อ่านสลิปจริง
> - เก็บข้อมูลลง Database จริง (Supabase PostgreSQL)
> - Deploy บน Vercel เข้าใช้งานผ่านเว็บได้จริง
> - มีระบบ Authentication จริง (OTP เบอร์โทรศัพท์)
> - สามารถเปิด Demo ให้กรรมการทดลองสแกนสลิปจริงได้เลยตอนนี้ครับ"

## 🔥 หมวด 4: คำถามเกี่ยวกับ Business Model

### ❓ "จะหาเงินจากไหน? ใครเป็นคนจ่าย?"
> **ตอบ:** "ใช้โมเดล **Freemium** ครับ:
> - **ฟรี:** สแกนสลิปได้จำกัดจำนวนต่อเดือน เหมาะกับร้านค้าขนาดเล็กมากที่มีธุรกรรมไม่กี่รายการ
> - **Premium (Subscription รายเดือน/ปี):** สแกนไม่จำกัด, AI วิเคราะห์เชิงลึก, Export รายงานเป็น Excel/PDF สำหรับทำบัญชีหรือยื่นภาษี
> - ผู้จ่ายคือเจ้าของร้านค้าเองครับ เพราะเราช่วยเขาประหยัดค่าจ้างนักบัญชี (ซึ่งแพงกว่ามาก) ครับ"

### ❓ "ราคาเท่าไหร่ถึงจะเหมาะกับ SME?"
> **ตอบ:** "จากการรีเสิร์ช แม่ค้าออนไลน์ยอมจ่ายได้ประมาณ 99-199 บาท/เดือน ถ้าเทียบกับค่าจ้างนักบัญชี 2,000-5,000 บาท/เดือน ของเราถูกกว่าหลายเท่า และคุ้มค่ากว่ามากครับ"

## 🔥 หมวด 5: คำถามเกี่ยวกับ Privacy & Security

### ❓ "ข้อมูลการเงินของผู้ใช้ปลอดภัยไหม? ส่งสลิปให้ AI อ่าน ข้อมูลรั่วไหลได้หรือเปล่า?"
> **ตอบ:** "เรื่องนี้สำคัญมากครับ เราดูแลเรื่องนี้หลายชั้น:
> 1. **Database:** ใช้ Supabase ซึ่งมี Row Level Security (RLS) ผู้ใช้แต่ละคนเห็นแค่ข้อมูลของตัวเองเท่านั้น
> 2. **AI Processing:** รูปสลิปถูกส่งไปประมวลผลกับ OpenAI API แล้วจะไม่ถูกเก็บไว้ใน AI Model (OpenAI ไม่ได้ใช้ข้อมูล API ไปเทรนโมเดลครับ)
> 3. **Authentication:** ใช้ OTP เบอร์มือถือ ไม่เก็บรหัสผ่าน ลดความเสี่ยงเรื่อง Password Breach
> 4. **ในอนาคต:** สามารถต่อยอดเป็นระบบ On-premise หรือ Private Cloud สำหรับลูกค้ากลุ่ม Enterprise ที่ต้องการความปลอดภัยสูงสุดได้ครับ"

## 🔥 หมวด 6: คำถามเกี่ยวกับ Challenge Adaptability (20%)

### ❓ "ถ้าเราเพิ่ม Surprise Condition เช่น 'ให้รองรับบิลกระดาษ' ทำได้ไหม?"
> **ตอบ:** "ทำได้เลยครับ เพราะ AI Vision ของเราอ่านได้ทั้งสลิปดิจิทัลและถ่ายรูปจากกระดาษจริง เราแค่ปรับ Prompt ให้ AI รู้จักรูปแบบบิลกระดาษเพิ่มเติม ไม่ต้องแก้โค้ดหลักเลยครับ"

### ❓ "ถ้าต้องเพิ่มระบบภาษี?"
> **ตอบ:** "ทำได้ครับ เพราะเรามีข้อมูลธุรกรรมแยกหมวดหมู่อยู่แล้วในฐานข้อมูล เราแค่เพิ่มหน้า Export ที่สรุปรายจ่ายตามหมวดภาษีที่ สรรพากรกำหนด แล้ว Generate เป็นรายงาน PDF ได้เลยครับ"

### ❓ "ถ้าต้องรองรับร้านค้าหลายสาขา?"
> **ตอบ:** "ใน Database Schema ของเรา ธุรกรรมทุกรายการผูกกับ User ID อยู่แล้ว เราแค่เพิ่ม Branch ID เข้าไปอีกชั้น แล้วทำหน้า Dashboard ที่ดูข้ามสาขาได้ ไม่ต้อง Refactor โค้ดหลักครับ"

## 🔥 หมวด 7: คำถามเกี่ยวกับ การต่อยอด (Future Plan)

### ❓ "มีแผนพัฒนาต่อยังไง?"
> **ตอบ:** "มี 4 แผนหลักครับ:
> 1. **Tax Integration:** สรุปเอกสารยื่นภาษีอัตโนมัติ
> 2. **SME Credit Score:** นำ Transaction History ไปสร้างโปรไฟล์ด้านการเงินให้ SME เพื่อใช้ยื่นกู้ธนาคาร (เพราะปกติ SME เล็กๆ ไม่มีงบดุลไปยื่นกู้)
> 3. **Gamification:** ระบบ "สู้บอสรายจ่าย" สร้างแรงจูงใจให้ผู้ใช้มีวินัยทางการเงิน
> 4. **Multi-Branch:** รองรับเจ้าของธุรกิจที่มีหลายสาขาครับ"

### ❓ "ทำไมไม่ทำ Feature X ตั้งแต่แรก?"
> **ตอบ:** "เราเชื่อในหลัก 'Simple but Valuable' ตามธีมของงานครับ MVP ต้องโฟกัสที่ปัญหาหลักก่อน ซึ่งก็คือ 'ทำให้ร้านค้ารู้กำไร-ขาดทุนจริง' ได้ เมื่อ Core Product แข็งแรงแล้ว ค่อยต่อยอดฟีเจอร์อื่นๆ ครับ"

## 🔥 หมวด 8: คำถามกวนๆ ที่กรรมการชอบถาม

### ❓ "ถ้าสลิปเบลอหรือถ่ายไม่ชัดล่ะ?"
> **ตอบ:** "ระบบมี Confidence Score ครับ ถ้า AI ไม่มั่นใจจะแจ้งเตือนให้ผู้ใช้ตรวจสอบช่องนั้นเป็นพิเศษ ผู้ใช้สามารถแก้ไขได้ก่อนกดบันทึก ไม่มีทางที่ข้อมูลจะเข้าฐานข้อมูลโดยไม่ผ่านการ Review ครับ"

### ❓ "ทำไมไม่ดึงข้อมูลจาก Mobile Banking API ตรงๆ เลย?"
> **ตอบ:** "ธนาคารในไทยยังไม่เปิด Open Banking API ให้นักพัฒนาทั่วไปครับ ดังนั้นการอ่านจากสลิปจึงเป็นวิธีเดียวที่ทำได้จริงตอนนี้ แต่ในอนาคตหากมี Open Banking เราพร้อมเชื่อมต่อทันทีครับ"

### ❓ "คู่แข่งของคุณคือใคร?"
> **ตอบ:** "มีแอปบัญชีอย่าง FlowAccount, PEAK ซึ่งเน้น SME ขนาดกลางขึ้นไป และ Money Tracker ทั่วไปที่ต้องคีย์เอง จุดแตกต่างของเราคือ:
> 1. **ไม่ต้องคีย์อะไรเลย** (Zero-Friction) คู่แข่งทุกตัวยังต้องพิมพ์
> 2. **เน้นกลุ่ม Micro SME** ที่คู่แข่งมองข้าม (ร้านตลาด ร้านชำ)
> 3. **AI ให้ Insights** ไม่ใช่แค่บันทึกตัวเลข แต่ช่วยวิเคราะห์และชี้ว่าต้องลดค่าใช้จ่ายตรงไหนครับ"

### ❓ "ถ้าผู้ใช้ไม่มีสลิปล่ะ? เช่นจ่ายเงินสดล่ะ?"
> **ตอบ:** "เรามีโหมด 'กรอกข้อมูลเอง' ด้วยครับ ผู้ใช้สามารถกรอกมือได้เลยสำหรับรายจ่ายเงินสด เพื่อให้ Dashboard ครบถ้วนที่สุดครับ"
